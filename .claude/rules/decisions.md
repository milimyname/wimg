Embeddings kept for smart categorization only (tx↔tx same-domain comparison works well). |

| Mar 2026 | Pure Zig FinTS over AqBanking               | AqBanking can't compile to WASM. Pure Zig implementation gives full control over FinTS 3.0 protocol, C ABI for iOS, no external deps. ~2000 lines for fints.zig + mt940.zig + fints_http.zig. |
| Mar 2026 | Bare FinTS envelope (no HNVSK/HNVSD)        | Comdirect rejects security envelope segments with `9130:unzulaessig`. PIN/TAN mode works with bare HNHBK/HNHBS wrapper. Some banks may require HNVSK — will need bank-specific flag if encountered. |
| Mar 2026 | FinTS static buffers over stack alloc        | Base64 encode/decode buffers (131KB each) in fints_http.zig cause stack overflow on iOS GCD threads (512KB default). File-level `struct { var }` statics solve it. iOS fetch dispatched on custom 2MB-stack Thread. |
| Mar 2026 | FinTS HTTP via C ABI callback                | Zig's `std.http.Client` works on native but not WASM. C ABI callback (`http_callback_fn`) lets iOS use URLSession, keeps transport platform-specific while protocol stays pure Zig. |
| Mar 2026 | HKKAZ v5 as default FinTS statement version  | Comdirect supports v5 (Account2 DEG). Different banks support v5/v6/v7 with different account formats (Account2/Account3/KTI1-IBAN). Version selection should come from BPD parsing. |
| Mar 2026 | FinTS DEG colons must not be escaped          | `280:BLZ` is a structural DEG (Data Element Group) separator. Passing it through `escapeFintsValue()` produced `280?:BLZ` which banks reject. Dedicated `writeHkidn()` helper writes DEGs directly. |
| Mar 2026 | TAN submit must carry task reference         | HKTAN process-2 needs `task_reference` from prior HITAN challenge; missing or wrong field mapping causes `9120/9110` and rejected TAN submit. |
| Mar 2026 | Preserve dialog state across HNVSD recursion | Parsing nested HNVSD must not reset outer parsed values (dialog_id/system_id). Resetting inside recursive parse led to `dialog_id=0` and `Initialisierung fehlt`. |
| Mar 2026 | Distinguish HITANS from HITAN strictly       | Prefix overlap (`HITANS` vs `HITAN`) can create fake TAN challenges and invalid photoTAN payload handling. Segment dispatch must match `HITANS:` before `HITAN:`. |
| Mar 2026 | HIKAZ statements require touchdown paging    | Response code `3040` carries continuation token; first page often caps around 100 transactions. Continue HKKAZ requests with touchdown until no token remains. |
| Mar 2026 | MT940 normalization needed for bank variants | Some streams require `@@ -> CRLF`, `-0000 -> +0000`, CR-only line handling, and optional funds code in `:61:`. Without this, valid payload parses as zero transactions. |
| Mar 2026 | Clear stale TAN state after success          | If successful responses still leave old challenge/hhduc in session, UI re-prompts TAN incorrectly. Reset `has_pending_tan` and challenge buffers when no new TAN request exists or MT940 is present. |
| Mar 2026 | HKKAZ version negotiated from BPD             | Parse `HIKAZS` segments in BPD to discover highest supported HKKAZ version (v5/v6/v7). Different banks use different account DEG formats per version. Fallback to v5 if no HIKAZS. |
| Mar 2026 | Transparent MT940 → CAMT fallback             | When `HIKAZS` absent but `HICAZS` present, use `HKCAZ` instead of `HKKAZ`. `camt.zig` parser handles Ntry extraction. Same Transaction model for both paths. |
| Mar 2026 | Decoupled TAN via HKTAN process-S polling     | Banks with pushTAN/appTAN use decoupled flow: HITAN returns process variant "S", app polls with HKTAN process-S (empty TAN, no PIN). BPD timings control poll intervals. |
| Mar 2026 | HKTAB for TAN medium selection                | Banks with `description_required=2` and multiple TAN media need explicit medium name in HKTAN process-4. HKTAB:3:4+0+A' fetches media list, HITAB returns names. Selected medium stored in session. |
| Mar 2026 | Structured 3040 touchdown extraction          | Prefer `parameter` field from ResponseCode over `text` for 3040 tokens. Text field may contain localized messages wrapping the actual token. Normalize by stripping after `:` or `=`. |
| Mar 2026 | Bank catalog drift checker over manual sync   | `scripts/check-bank-drift.ts` compares official FinTS institute CSV against banks.zig. Reports missing/changed/removed banks. Run manually — official CSV arrives irregularly. |
| Mar 2026 | Top-bank matrix for protocol regression       | `scripts/test-bank-matrix.ts` probes 8 banks with anon init. Catches wire/envelope regressions early. Deutsche Bank + Postbank reject `9110` — likely need envelope or bank-family flags. |
| Mar 2026 | All scripts in Bun, not Python                | Rewrote `test-bank-matrix` and `check-bank-drift` from Python to TypeScript (Bun). Deleted 10 one-off Python iteration scripts (`test-fints*.py`). Eliminates Python + `requests` as a dependency — toolchain is now Zig + Bun only. Bun's built-in `fetch()` replaces `requests`. Bank matrix produces identical results. |
| Mar 2026 | Anonymous customer_id=9999999999 per spec      | FinTS spec and python-fints both use `9999999999` for anonymous access, not `0`. Some banks (Deutsche Bank, Postbank) reject `0`. Changed in both Zig (`buildAnonInit`) and bank matrix script. |
| Mar 2026 | Bank-family detection from URL                 | `BankFamily` enum + `detectBankFamily()` in `banks.zig` identifies Deutsche Bank/Postbank/norisbank by URL pattern. Avoids flagging ~301 individual entries — cleaner than per-entry flags. Auto-set on `FintsSession` during init. |
| Mar 2026 | HNVSK/HNVSD envelope for anonymous init        | Deutsche Bank + Postbank reject bare anonymous init with `9110`. `buildAnonInitWithEnvelope` wraps HKIDN+HKVVB in dummy HNVSK/HNVSD (PIN:1, sec_func=998, user=9999999999). Standard banks (Comdirect, Sparkasse) continue using bare envelope. |
| Mar 2026 | R2 JSON blob for sync (potential DO SQLite migration) | Current sync stores a single JSON blob per sync key in R2. Every push reads + rewrites the entire blob — O(n) per operation. Works fine at current scale (~1K-10K transactions). DO SQLite would give indexed upserts (O(1) per row), no extra binding, and no max-size risk. Not worth migrating now (working system, small data), but DO SQLite is the better pattern for any new sync work or if scale becomes an issue. |
